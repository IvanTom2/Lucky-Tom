## 1. Язык и фреймворк

### Python + FastAPI

**Выбор:** Python 3.12+, FastAPI.

**Обоснование:**
- Зафиксировано в технических предпочтениях проекта.
- FastAPI — нативная поддержка async, автогенерация OpenAPI, богатая экосистема.
- Скорость разработки критична для MVP за 2–4 недели.
- Узкое место highload будет в БД и IO, не во фреймворке.

**gRPC:** grpcio + grpcio-tools — стандартный подход для Python, генерация клиентов и серверов из `.proto` файлов.

**Применяется в:** Core-сервис, Slot-сервисы, Analytics-сервис.

**Trade-off:** Не самый производительный Python фреймворк, но для большинства highload сценариев достаточно. Litestar быстрее, но меньше экосистема и community.

**После MVP:** при необходимости экстремальной производительности — рассмотреть Litestar как замену без изменения бизнес-логики (hexagonal упростит замену адаптера).

## 2. Базы данных

### 2.1 PostgreSQL — основное хранилище

**Применяется в:** Core-сервис, Slot-сервисы.

**Что хранит:**

*Core-сервис:*
- User, Transaction — финансовые операции, ACID обязателен.
- RegisteredSlot, RegisteredSlotVersion, RegisteredSlotActiveVersion — реляционные данные, редкие записи.
- RegisteredSlotVersionSymbol, RegisteredSlotVersionStatistics — облегчённые копии данных слота.
- Constraints — статистические ограничения.

*Slot-сервис:*
- Versions, VersionConfig — иммутабельные данные версий.
- SymbolVersion, SymbolVersionUI, SymbolVersionPayout, SymbolVersionConfig — данные символов.
- VersionStatistics — статистика версий.
- RegisteredClient — зарегистрированные клиенты.

**ORM:** SQLAlchemy async + Alembic (миграции).

**Обоснование:**
- Полный ACID — критично для баланса пользователя (Strong Consistency).
- Нативная поддержка JSON — удобно для `extra: dict` в SymbolVersionConfig.
- Зрелая экосистема в Python, asyncpg под капотом SQLAlchemy async.
- Read replicas для масштабирования чтений при необходимости.

**Trade-off:** при очень высоком write throughput VersionSpinStatistics — узкое место. Решено выносом в ClickHouse.

---

### 2.2 ClickHouse — аналитическое хранилище

**Применяется в:** Analytics-сервис.

**Что хранит:**
- VersionSpinStatistics — реальная статистика спинов (user, bet, win, bonus_game).

**Обоснование:**
- Столбцовая БД, оптимизирована под append-only write и аналитические запросы.
- Высокий write throughput — каждый спин генерирует запись.
- Эффективные агрегации для подсчёта статистики по спинам.

**Trade-off:** отдельный инфраструктурный компонент. Для MVP оправдано — проектируем правильно сразу, не создаём технический долг.

**После MVP:** если throughput спинов вырастет до экстремальных значений — ClickHouse масштабируется горизонтально.

---

### 2.3 Redis — кэш и сессии

**Применяется в:** Core-сервис.

**Что хранит:**
- Активная версия слота — инвалидируется при смене версии.
- Данные о слотах — инвалидируется при обновлении.
- Game Session — короткоживущие данные с TTL (таймаут сессии).

**Клиент:** aioredis (async).

**Обоснование:**
- Стандарт индустрии для кэширования.
- TTL — идеален для игровых сессий с таймаутом.
- При нескольких инстансах Core — централизованная инвалидация кэша.
- Разгружает PostgreSQL от частых read/write операций сессий.

**Стратегия инвалидации кэша:**
- Активная версия слота: явная инвалидация при смене версии (write-through: запись в БД → обновление кэша).
- Game Session: TTL — автоматическое завершение по таймауту бездействия.

**Trade-off:** механизм атомарного обновления кэша и БД при смене активной версии — технический долг, требует проработки.

## 3. Брокер сообщений

### RabbitMQ + aio-pika

**Применяется для:**
1. **Slot → Core:** события о новых версиях слота (низкий throughput, редко).
2. **Slot → Analytics-сервис:** события о спинах (высокий throughput, каждый спин).

**Клиент:** aio-pika — async обёртка над RabbitMQ, хорошая интеграция с FastAPI.

**Обоснование:**
- Знакомый инструмент — снижает риски MVP.
- At-least-once доставка из коробки.
- Персистентность очередей — сообщения не теряются при рестарте.
- Достаточный throughput для MVP сценария.

**Конфигурация:**
- Отдельный exchange для версий слотов (fanout — Core подписан).
- Отдельный exchange для событий спинов (fanout — Analytics подписан).
- Durable queues — персистентность сообщений.

**Trade-off:** при экстремальном throughput спинов RabbitMQ может стать узким местом. Hexagonal архитектура упростит замену на Kafka в будущем — замена только адаптера, бизнес-логика не меняется.

**После MVP:** рассмотреть Kafka для потока событий спинов если RabbitMQ не справляется с throughput.

## 4. Инфраструктура

### Docker + Docker Compose

**Обоснование:**
- Open source проект — простота развёртывания критична.
- Один `docker-compose.yml` поднимает всю систему одной командой.
- Идеальный Developer Experience для новых контрибьюторов.

**Состав docker-compose:**
- Core-сервис
- Slot-сервис (пример эталонного слота)
- Analytics-сервис
- PostgreSQL (для Core и Slot — отдельные БД)
- ClickHouse
- RabbitMQ
- Redis

### CI/CD: GitHub Actions

**Обоснование:**
- Бесплатно для open source проектов.
- Простая интеграция с GitHub.
- Большое количество готовых actions.

**Минимальный pipeline для MVP:**
- Lint + type check (ruff, mypy).
- Unit тесты.
- Build Docker образов.

**После MVP:** Kubernetes манифесты для production-grade оркестрации при необходимости.

## 5. Observability

### 5.1 Логирование: structlog (MVP)

**Обоснование:**
- Структурированные логи в JSON — удобно для парсинга и фильтрации.
- Вывод в stdout — Docker Compose собирает автоматически.
- Минимальная настройка, не отвлекает от разработки бизнес-логики.

**Применяется во всех сервисах.**

### 5.2 Трейсинг: OpenTelemetry + Jaeger (конец MVP или по необходимости)

**Обоснование:**
- OpenTelemetry — vendor-agnostic стандарт, инструментирование один раз.
- FastAPI и gRPC инструментируются почти автоматически.
- Jaeger поднимается одним контейнером в Docker Compose.
- Позволяет видеть полный путь запроса через Core → Slot → Core (flow спина).

**Trade-off:** откладывается до конца MVP или до реальной потребности в отладке распределённых запросов.

### 5.3 Мониторинг: Prometheus + Grafana (после MVP)

**После MVP:** метрики сервисов, дашборды, алёрты.
