**Версия:** 0.1
**Дата:** 2026-02-28
**Автор:** Tomilov Ivan

## 1. Цель продукта

Lucky-Tom — это open source система управления сетевыми игровыми автоматами, предназначенная исключительно для образовательных и развлекательных целей без коммерческой деятельности.

**Цель для Игрока:** симуляция игры в онлайн-казино с реальными условиями, но без реальных денег. Игрок может развлечься, проверить личные гипотезы о влиянии RTP, дисперсии и других параметров слота на результат — не тратя денег.

**Цель для Администратора:** творческий процесс создания собственных игровых автоматов, практика работы с распределёнными системами, возможность "повеселиться с точки зрения разработки".

**Принципиальное ограничение:** система не может и не должна использоваться как реальная казино-платформа. Использование реальных денег запрещено на законодательном уровне и является жёстким ограничением продукта.

## 2. Целевая аудитория и роли

### Целевая аудитория
- Разработчики, которые хотят практиковаться в создании распределённых систем и геймблинг-алгоритмов.
- Люди, которые хотят взглянуть на систему с обеих сторон: как игрок и как разработчик.
- Проект публикуется в открытом доступе (open source). Централизованного хостинга нет — каждый разворачивает систему самостоятельно.

### Роли

| Роль | Описание | Ключевые действия |
|------|----------|-------------------|
| Игрок | Зарегистрированный пользователь онлайн-слотов | Зарегистрироваться, пополнить баланс, вывести средства, найти слот, подключиться, сделать спин |
| Администратор | Разработчик, создающий и конфигурирующий слоты | Создать слот, создать версию слота, менять активную версию в run-time, останавливать и активировать слот |

**Ограничения ролей:**
- Роли строго разделены: один человек не может совмещать роль игрока и администратора.
- Чтобы стать игроком — нужно отдельно зарегистрироваться.
- Администратор — это разработчик. Учётная запись создаётся при инициализации сервиса.

## 3. Ключевые сценарии использования (высокоуровнево)

**Сценарий 1: Игрок играет в слот**
Пользователь регистрируется, пополняет мок-баланс, находит доступный слот, подключается к нему и делает спины. По итогу баланс меняется в зависимости от результатов игры. При завершении сессии (или таймауте) сессия закрывается. При следующем подключении начинается новая сессия.

**Сценарий 2: Администратор создаёт и запускает слот**
Разработчик реализует слот как микросервис на основе готовых компонентов системы. Создаёт версию слота с символами, таблицами выплат и конфигурацией. Система автоматически рассчитывает статистику. После успешной проверки статистики регистрирует слот в основной системе и выбирает активную версию.

**Сценарий 3: Администратор меняет версию слота в run-time**
Администратор создаёт новую версию слота с изменёнными параметрами. После расчёта статистики и прохождения ограничений переключает активную версию — логика расчётов меняется без остановки сервиса.

**Сценарий 4: Администратор останавливает слот**
Администратор деактивирует слот. Текущие спины доигрываются до конца, новые подключения не принимаются. При необходимости слот активируется обратно.

## 4. Метрики успеха

Метрики успеха носят технический характер:

- Разработчик может реализовать интерфейсы, быстро поднять слот-сервис — и он работает надёжно и быстро даже под нагрузкой.
- Баланс пользователя изменяется надёжно и консистентно в распределённой системе.
- Администратор может переключить версию слота в run-time без остановки сервиса.
- Система выдерживает нагрузку в тысячи одновременных игроков (ориентир highload).

Бизнесовых метрик (DAU, retention, звёзды на GitHub и т.д.) на данном этапе нет.

## 5. Ограничения

### Временные
- MVP: 2–4 недели в режиме vibe-coding.

### Бюджетные
- Не определены.

### Юридические
- Использование реальных денег категорически запрещено.
- Система не может функционировать как коммерческая казино-платформа.

### Платформа и доступность
- Система распространяется как open source — централизованного хостинга нет.
- Бэкенд предоставляет REST API; клиентский интерфейс может быть любым.
- Развёртывание производится самостоятельно каждым пользователем.

### Баланс
- Минимальный депозит: 100 условных единиц.
- Максимальный депозит: 1 000 000 условных единиц.
- Минимальная сумма вывода: 100 условных единиц.
- Баланс не может быть меньше 0.

### Сессии
- Игровая сессия завершается по таймауту.
- Восстановление сессии после разрыва не поддерживается — при следующем подключении начинается новая сессия.

## 6. Scope / Вне scope

### В scope (MVP)
- Регистрация и авторизация игрока.
- Мок-баланс: пополнение и вывод средств.
- Создание слота как микросервиса на основе готовых компонентов.
- Создание и версионирование слотов с символами, таблицами выплат, конфигурацией.
- Автоматический расчёт статистики версии слота.
- Статистические ограничения для версий слота.
- Регистрация слота в основной системе.
- Выбор и смена активной версии слота в run-time.
- Поиск доступных слотов.
- Игровая сессия: подключение к слоту, спины, изменение баланса.
- Остановка и активация слота.
- Готовые компоненты для быстрой сборки слот-сервисов.

### Вне scope (после MVP)
- UI / фронтенд (SPA-приложение — планируется после MVP).
- История транзакций и спинов для игрока.

### Вне scope (навсегда)
- Платёжные системы и реальные деньги.
- Мультиязычность.
- Централизованный хостинг.
- Мониторинг.
- Мультивалютность (баланс — просто число без привязки к валюте).

## 7. Открытые вопросы

- Что происходит с активными игровыми сессиями в момент остановки слота — они доигрываются до конца текущего спина или обрываются немедленно? *(предварительно: доигрываются до конца спина — требует уточнения)*
- Какой таймаут игровой сессии считается приемлемым?
- Механизм консистентности кэша и БД при смене активной версии слота не определён.
- Механизм идемпотентности запроса на спин не определён.
- Механизм предотвращения дублирующейся регистрации клиента не определён.
- Механизм создания и управления учётной записью администратора не определён.

## 8. Технические предпочтения

*(Зафиксированы отдельно — не являются бизнес-требованиями)*

- Язык и фреймворк: Python, FastAPI.
- Межсервисное взаимодействие: gRPC.
- Архитектурный стиль слот-сервиса: Hexagonal Architecture.
- Клиентский интерфейс: REST API (бэкенд не диктует тип клиента).
- Кэширование: активная версия слота и данные о слотах кэшируются для быстрого доступа.
- Консистентность баланса: Strong Consistency.
- Код: асинхронный.
