# System Design: Lucky-Tom

> Этот документ описывает архитектуру системы на уровне компонентов, взаимодействий и trade-offs.
> Формируется на основе system-vision.md и business-requirements.md.

**Версия:** 0.1
**Дата:** 2026-02-28
**Автор:** Tomilov Ivan

## 1. Обзор архитектуры

Lucky-Tom — распределённая система, состоящая из одного Core-сервиса и множества независимых Slot-сервисов. Внутренняя архитектура каждого сервиса — **Hexagonal (Ports & Adapters)**, что обеспечивает изоляцию бизнес-логики и упрощает будущее разделение или замену компонентов.

### Ключевые архитектурные решения

| Решение | Выбор | Обоснование |
|---------|-------|-------------|
| Core-сервис | Монолит | MVP за 2–4 недели. Hexagonal внутри позволит разделить на сервисы в будущем без переписывания логики. |
| Slot-сервисы | Независимые сервисы | По природе системы: каждый слот — отдельная игровая логика, отдельный деплой. |
| Синхронизация версий | Асинхронно через очередь | Slot и Core слабо связаны. Очередь гарантирует доставку при временной недоступности Core. |
| Консистентность баланса | Strong Consistency (CP) | Баланс не может уйти в минус. Лучше вернуть ошибку, чем дать неконсистентный результат. |
| Консистентность версий | Eventual Consistency (AP) | Административные данные. Небольшая задержка синхронизации приемлема. |

## 2. Компоненты системы

### 2.1 Core-сервис

**Тип:** Монолит, Hexagonal-архитектура внутри.
**Интерфейс для клиентов:** REST API.
**Интерфейс для Slot-сервисов:** gRPC (входящие запросы на изменение баланса).

Содержит три логических домена:

**User & Balance**
- Регистрация и авторизация игроков.
- Управление мок-балансом: пополнение, вывод.
- Хранение истории транзакций.
- Обработка gRPC запросов от Slot-сервисов на списание ставки и начисление выигрыша.

**Slot Management**
- Регистрация Slot-сервисов (host, port, token).
- Хранение облегчённой копии данных о версиях слотов (название, статистика, символы для отображения).
- Управление активной версией слота.
- Статистические ограничения (Constraints) для версий.
- Приём событий о новых версиях из очереди сообщений.

**Game Session**
- Управление игровыми сессиями (создание, таймаут, завершение).
- Ограничение: одна активная сессия на игрока.
- Проксирование подключения игрока к Slot-сервису.

---

### 2.2 Slot-сервис

**Тип:** Независимый сервис, Hexagonal-архитектура внутри.
**Интерфейс для клиентов:** WebSocket (игровая сессия).
**Интерфейс для Core:** gRPC (исходящие запросы на изменение баланса).

Ответственности:
- Реализация игровой логики и геймблинг-алгоритмов.
- Управление версиями слота: символы, таблицы выплат, конфигурация.
- Расчёт статистики версий (RTP, дисперсия, hit-rate и др.).
- Обработка запросов на спины: идемпотентность, вычисление результата.
- Публикация событий о новых версиях в очередь сообщений.
- Хранение реальной статистики спинов (VersionSpinStatistics).
- Регистрация Core-сервиса как клиента (RegisteredClient).

---

### 2.3 Брокер сообщений

**Тип:** Очередь сообщений (абстракция).
**Назначение:** Асинхронная доставка событий о новых версиях слота от Slot-сервиса к Core-сервису.

Требования к очереди:
- At-least-once доставка.
- Сохранение порядка сообщений (FIFO в рамках одного слота).
- Персистентность сообщений (не теряет события при рестарте).

## 3. Взаимодействие между компонентами

### 3.1 Карта взаимодействий

```
Клиент
  │
  ├─── REST API ──────────────► Core-сервис
  │                                  │
  │                                  ├─ gRPC (регистрация, смена версии) ──► Slot-сервис
  │                                  │
  │                                  ├─ gRPC (входящий: баланс) ◄─────────── Slot-сервис
  │                                  │
  │                                  └─ Подписка на очередь ◄── Брокер ◄─── Slot-сервис
  │
  └─── WebSocket ─────────────► Slot-сервис
```

### 3.2 Детали взаимодействий

| Направление | Протокол | Тип | Назначение |
|-------------|----------|-----|------------|
| Клиент → Core | REST API | Синхронный | Регистрация, авторизация, баланс, поиск слотов |
| Клиент ↔ Slot | WebSocket | Постоянное соединение | Игровая сессия, спины |
| Core → Slot | gRPC | Синхронный с таймаутом | Регистрация слота, смена активной версии |
| Slot → Core | gRPC | Синхронный | Списание ставки, начисление выигрыша (адрес: `balance_webhook`) |
| Slot → Core | Очередь сообщений | Асинхронный, at-least-once | Уведомление о новых версиях слота |

## 4. Flow спина (детально)

```
Клиент                    Slot-сервис                Core-сервис
  │                            │                           │
  │── запрос на спин (WS) ───► │                           │
  │                            │ проверка идемпотентности  │
  │                            │ (нет незавершённого спина)│
  │                            │                           │
  │                            │── gRPC: списать ставку N ►│
  │                            │                           │ проверка баланса
  │                            │                           │ запись транзакции -N
  │                            │◄─ ok / insufficient ──────│
  │                            │                           │
  │                            │ вычисление результата     │
  │                            │                           │
  │                            │── gRPC: начислить M ──────►│
  │                            │                           │ запись транзакции +M
  │                            │◄─ ok ─────────────────────│
  │                            │                           │
  │◄── результат спина (WS) ───│                           │
```

**Важные ограничения:**
- Одна активная сессия на игрока — исключает параллельные спины между сессиями.
- Идемпотентность внутри сессии — новый спин невозможен пока не завершён предыдущий.
- Ставка списывается до вычислений (MVP). Если Slot упал после списания — ставка потеряна. Осознанный trade-off: простота vs честность.

## 5. Flow регистрации слота

```
Администратор             Core-сервис                Slot-сервис
  │                            │                           │
  │── указать host:port ──────►│                           │
  │                            │── gRPC: register ────────►│
  │                            │                           │ сохранить RegisteredClient
  │                            │◄─ ok (token) ─────────────│
  │                            │ сохранить RegisteredSlot  │
  │◄─ ok ──────────────────────│                           │
```

**Если Slot недоступен:** немедленная ошибка администратору — слот не регистрируется.

## 6. Flow синхронизации версий

```
Slot-сервис               Брокер сообщений           Core-сервис
  │                            │                           │
  │ создана новая версия        │                           │
  │── событие (версия + seq) ──►│                           │
  │                            │── доставка ──────────────►│
  │                            │                           │ сохранить облегчённую
  │                            │                           │ копию версии
```

**Гарантии:**
- At-least-once доставка — событие не потеряется.
- Порядковый номер (`sequence_number`) — для корректного порядка обработки.
- Идемпотентность обработки — версии иммутабельны, повторная обработка безопасна.
- Если очередь недоступна при публикации — немедленная ошибка администратору.

## 7. Модель данных и владение

### 7.1 Владение данными

| Данные | Владелец | Копия | Тип консистентности |
|--------|----------|-------|---------------------|
| User, Transaction | Core | — | Strong Consistency (CP) |
| RegisteredSlot, RegisteredSlotActiveVersion | Core | — | Strong Consistency (CP) |
| RegisteredSlotVersion, RegisteredSlotVersionSymbol, RegisteredSlotVersionStatistics | Core | Облегчённая копия из Slot | Eventual Consistency (AP) |
| Versions, VersionConfig, SymbolVersion, SymbolVersionUI, SymbolVersionPayout, SymbolVersionConfig, VersionStatistics, Constraints | Slot | — | Strong Consistency (CP) |
| VersionSpinStatistics | Slot | — | Strong Consistency (CP) |
| RegisteredClient | Slot | — | Strong Consistency (CP) |

### 7.2 Кэширование

| Данные | Где кэшируется | Инвалидация |
|--------|---------------|-------------|
| Активная версия слота | Core | При смене активной версии: запись в БД → обновление кэша |
| Данные о слотах | Core | При обновлении данных слота |

**Trade-off кэша:** механизм консистентности кэша и БД при смене активной версии окончательно не определён — технический долг.

## 8. Отказоустойчивость

### 8.1 Сценарии отказов

| Сценарий | Поведение | Механизм |
|----------|-----------|----------|
| Core недоступен во время спина | Slot возвращает ошибку игроку | Circuit Breaker + ретраи на стороне Slot |
| Slot упал после списания ставки | Ставка потеряна (MVP) | Осознанный trade-off. Резервация — технический долг |
| Slot упал во время сессии | Сессия завершается по таймауту | Таймаут бездействия сессии |
| Очередь недоступна при публикации версии | Ошибка администратору | Немедленный отказ. Transactional Outbox — технический долг |
| Попытка второго спина до завершения первого | Отказ | Идемпотентность на уровне Slot-сервиса |
| Попытка второй сессии | Отказ | Ограничение: одна активная сессия на игрока |

### 8.2 Circuit Breaker (Slot → Core)

- Slot отслеживает неудачные gRPC запросы к Core.
- До открытия Circuit Breaker: N ретраев с интервалом.
- После N неудач подряд: Circuit Breaker открывается, Slot сразу возвращает ошибку без ожидания таймаута.
- При восстановлении Core: Circuit Breaker закрывается, работа возобновляется.

## 9. CAP и Trade-offs

### 9.1 Теорема CAP

| Подсистема | Выбор | Обоснование |
|------------|-------|-------------|
| Баланс пользователя | **CP** (Consistency + Partition tolerance) | Баланс не может быть неконсистентным. При проблемах с сетью — ошибка, не грязные данные. |
| Версии слотов в Core | **AP** (Availability + Partition tolerance) | Core остаётся доступным даже при недоступности очереди. Данные о версиях могут быть немного устаревшими — приемлемо для административных данных. |

### 9.2 Ключевые trade-offs

| Trade-off | Выбор в MVP | Альтернатива (после MVP) |
|-----------|-------------|--------------------------|
| Простота vs честность при падении Slot | Ставка списывается до вычислений. При падении Slot — ставка теряется. | Резервация ставки через таблицу SpinReservation. |
| Простота vs надёжность очереди | Немедленная ошибка при недоступности очереди. | Transactional Outbox на стороне Slot-сервиса. |
| Простота vs observability | Health check Slot-сервисов не реализован. Сессии завершаются по таймауту. | Активный Health Check + автоматическое завершение сессий при падении Slot. |
| Монолит vs микросервисы (Core) | Монолит с Hexagonal внутри. | Разделение на независимые сервисы по доменам без переписывания логики. |

## 10. Технический долг

| Пункт | Приоритет | Описание |
|-------|-----------|----------|
| Резервация ставки | Высокий | SpinReservation: зарезервировать ставку до вычислений, применить итог после. Защищает игрока при падении Slot. |
| Transactional Outbox | Средний | Надёжная публикация событий версий при недоступности очереди. |
| Health Check Slot-сервисов | Средний | Активное детектирование падения Slot + автоматическое завершение сессий. |
| Консистентность кэша и БД | Высокий | Механизм атомарного обновления кэша и БД при смене активной версии. |
| Механизм управления администратором | Низкий | Сейчас учётная запись создаётся при инициализации. Нужен полноценный механизм. |
| Предотвращение дублирующейся регистрации клиента | Средний | Защита от повторной регистрации одного и того же Core-сервиса в Slot. |

---

## 11. Внешние интеграции

Внешних интеграций нет. Система полностью замкнута.

- Платёжные системы — вне scope навсегда.
- `balance_webhook` — gRPC адрес Core-сервиса для входящих запросов от Slot (внутренняя интеграция).
