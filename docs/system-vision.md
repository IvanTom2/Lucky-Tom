## 1. Что это за продукт и зачем он нужен

### Суть продукта
- Lucky-Tom - это система управления сетевыми игровыми автоматами. 
- Предназначена для создания и быстрого развертывания онлайн-слотов в демонстративных целях (без коммерческой деятельности).
- Демонстрирует варианты геймблинг-алгоритмов и управления множеством игровых машин. 

### Целевая аудитория
- Разработчики и люди, которые хотят взглянуть на систему с разных сторон: побыть как игроком (играть "в казино" на нереальные деньги), так и разработчиком (создавать и конфигурировать слоты).

### Ключевая ценность
- Систему можно использовать для демонастративного примера финансовой уязвимости пользователей онлайн-слотов. 
- Пользователь системы может быстро создавать и настраивать слоты под любую тематику.
- Пользователь системы может конфигурировать слоты прямо в run-time, меняя правила игры и влияя на финансовые показатели слота. 

### Критерии успеха
- Продукт позволяет зарегистрировать пользователя и надёжно изменять его баланс в распределённой системе.
- Продукт позволяет быстро создавать и конфигурировать версии онлайн-слота, изменяя правила игры прямо в run-time.
- Разработчик может написать минимальное количество логики (реализовать интерфейсы и контракты), быстро поднять сервис  и он будет надёжно и быстро работать даже под нагрузкой.

## 2. Основной функционал

### Роли пользователей

| Роль | Описание | Ключевые действия |
|------|----------|-------------------|
| Игрок | Пользователь онлайн-слотов | Зарегистрироваться, сделать депозит, вывести средства, подключиться к слоту, сделать спин в слоте |
| Администратор | Создаёт и конфигурирует слоты | Создать слот, создать версию слота, изменить активную версию слота |

**Ограничения ролей:**
- Один человек не может совмещать роли игрока и администратора - это строго разделённые роли.
- Чтобы стать игроком - нужно отдельно зарегистрироваться как игрок.
- Администратор на текущем этапе - это разработчик. Учётная запись администратора создаётся при инициализации сервиса. Механизм управления администраторами пока не продуман и не является приоритетом для MVP.

### Фичи

**F-001: Регистрация пользователя**
- Кто: Пользователь
- Что: Пользователь регистрируется в системе, вводит username и password.  Система сохранят данные о нем, устанавливает ему баланс 0 и выдает ему сессионный токен. 
- Статус: MVP

**F-002: Пополнить баланс**
- Кто: Пользователь
- Что: Пользователь пополняет свой баланс. Вводит сумму - и увеличивает на нее баланс. Это мок-баланс, поэтому реального получения денег система не производит, просто увеличивает баланс. 
- Статус: MVP

**F-003: Вывести средства**
- Кто: Пользователь
- Что: Пользователь выводит свои средства. Вводит сумму - и уменьшает на нее баланс. Это мок-баланс, поэтому реального пополнения денег система не производит, просто уменьшает баланс. 
- Статус: MVP

**F-004: Создать слот**
- Кто: Администратор
- Что: Создает слот. Создать слот - это значит написать отдельный микросервис по заданным интерфейсам, возможно реализовать новые геймблинг-алгоритмы и механики в соответствии с интерфейсами. Каждый слот будет независимым сервисом. 
- Статус: MVP

**F-005: Создать версию слота**
- Кто: Администратор
- Что: Создать версию слота (название, описание). Версия слота - это набор символов (name) в рамках версии, а также их UI (name, image, designation), Payout (table: dict[int, Decimal] - таблица выплат), Config (proba, multiplier, is_wild, is_scatter, bonus_game_only). У версии слота есть Config (lines: list[int] - варианты выбора кол-ва линий, bets: list[Decimal] - варианты выбора ставки). 
- Статус: MVP

**F-006: Скопировать / частично создать версию слота**
- Кто: Администратор
- Что: Можно создать версию слота из предыдущих версий: скопировать определенные символы, или их UI и изменить Payout. В общем, при создании версии можно указывать сущности на копирование из `id`-версии. 
- Статус: MVP

**F-007: Создать статистику версии слота**
- Кто: Администратор
- Что: После создания версии слота запускается автоматический процесс расчета статистики. Статистика строго связана с версией слота. Версию слота нельзя редактировать - можно создать только новую. Соответственно, статистика будет отражать положение дел конфигурации версии. Если у версии слота не рассчитана статистика - ее нельзя выбрать активной (пока статистика не будет рассчитана). 
- Статус: MVP

**F-008: Добавить статистические ограничения слота**
- Кто: Администратор
- Что: Администратор может добавить статистические ограничения для слота. Это набор min-max значение статистики. Если статистический показатель версии не попадает в диапазон - ее невозможно выбрать в качестве активной версии. 
- Статус: MVP

**F-009: Регистрация слота в основной системе**
- Кто: Администратор
- Что: Администратор может зарегистрировать слот в системе. Что это значит. Наш слот - микросервис, к нему могут обращаться разные клиенты (условные "казино", как основная система управления пользователями). Основная система будет содержать базовую информацию о слоте: его название, версии, статистику и прочее. При регистрации создается токен, по которому будет производиться обмен информацией между сервисами. 
- Статус: MVP

**F-010: Выбрать версию слота в основной системе**
- Кто: Администратор
- Что: Администратор может в run-time сменить версию слота. Запросы начнут отправляться к соответствующей версии слота по тому же адресу, изменится только логика расчетов внутри слота. 
- Статус: MVP

**F-011: Поиск слотов**
- Кто: Пользователь
- Что: Пользователь получает список слотов, доступных к игре в рамках основного сервиса. 
- Статус: MVP

**F-012: Подключиться к слоту**
- Кто: Пользователь
- Что: Пользователь подключается к слоту и входит в игровую сессию. 
- Статус: MVP

**F-013: Сделать спин**
- Кто: Пользователь
- Что: Пользователь подключен к слоту. Выбирает ставку, количество линий и делает спин. В зависимости от результатов спина меняется его баланс. 
- Статус: MVP

**F-014: Остановить слот**
- Кто: Администратор
- Что: Полностью деактивировать возможность игры в слоте (на время). 
- Статус: MVP

**F-015: Активировать слот**
- Кто: Администратор
- Что: Отменить остановку слота.  
- Статус: MVP

### После MVP
- UI в виде SPA-приложения для работы с системой с точки зрения пользователя.

| Атрибут | Описание | Ограничения |
|---------|----------|-------------|
| username | Ник пользователя | Уникальное значение |

**Бизнес-правила:**
- Нельзя удалить пользователя. Его можно деактивировать.
- Деактивированный пользователь не может войти в систему и играть.

## 4. Связи между сущностями

**Core-сервис:**

| Сущность A | Связь | Сущность B | Описание |
|------------|-------|------------|----------|
| User | 1 → N | Transaction | Без User не может быть Transaction |
| RegisteredSlot | 1 → N | RegisteredSlotVersion | |
| RegisteredSlotVersion | 1 → N | RegisteredSlotVersionSymbol | |
| RegisteredSlotVersion | 1 → 1 | RegisteredSlotVersionStatistics | |
| RegisteredSlot | 1 → 0..1 | RegisteredSlotActiveVersion | У слота может быть 0 или 1 активная версия |

**Slot-сервис:**

| Сущность A | Связь | Сущность B | Описание |
|------------|-------|------------|----------|
| SlotVersion | 1 → N | SlotVersionStatistics | У версии слота может быть от 0 до N расчётов статистики |
| SlotVersion | 1 → 1 | SlotVersionConfig | У версии слота ровно 1 конфиг |
| SlotVersion | 1 → N | SymbolVersion | У версии слота может быть N версий символов |
| SymbolVersion | 1 → 1 | SymbolVersionUI | |
| SymbolVersion | 1 → 1 | SymbolVersionPayout | |
| SymbolVersion | 1 → 1 | SymbolVersionConfig | |

**Ключевые зависимости:**
- Пока что ничего нельзя удалять. Всё через soft_delete (deleted_at). По мере разработки правила могут измениться.

## 5. Порядок разработки

- Принцип: сначала самое рискованное, потом - всё остальное. 
- Каждая фаза должна давать работающий результат. 

### Фаза 1: Core-сервис: пользователь (MVP)
**Цель:** Можно создать пользователя и изменить ему баланс
**Фичи:** F-001, F-002, F-003
**Сущности:** User, Transaction
**Результат:** Создать пользователя, добавить ему баланс, снять его баланс. 

### Фаза 2: Slot-сервис: создать слот (MVP)
**Цель:** Можно создать слот. Он будет изолированно работать (сам с собой).
**Фичи:** F-004, F-005, F-006, F-007, F-008
**Сущности:** Versions, VersionConfig, VersionStatistics, SymbolVersion, SymbolVersionUI, SymbolVersionPayout, SymbolVersionConfig, Constraints
**Результат:** Можно создать и поднять слот. Внутри самого Slot-сервиса можно запускать спины, считать статистику и локально тестировать.

### Фаза 3: Slot-сервис: запросы на регистрацию клиента
**Цель:** Slot-сервис должен содержать контракты регистрации в сторонней системе. При обращении он регистрирует сервис и сохраняет данные для взаимодействия с ним.
**Фичи:** F-009
**Сущности:** RegisteredClient
**Результат:** Slot-сервис может принимать запросы на регистрацию клиента.

### Фаза 4: Core-сервис: зарегистрировать слот
**Цель:** Основная логика регистрации слота в системе. Реализация контрактов Slot-сервисов.
**Фичи:** F-009
**Сущности:** RegisteredSlot, RegisteredSlotVersion, RegisteredSlotActiveVersion, RegisteredSlotVersionSymbol, RegisteredSlotVersionStatistics
**Результат:** Core-сервис может отправлять запросы на регистрацию слота.

### Фаза 5: Core-сервис: выбор и поиск слота
**Цель:** Логика выбора версии Slot-сервиса. Логика получения списка слотов.
**Фичи:** F-010, F-011
**Сущности:** —
**Результат:** Можно менять версии слотов. Пользователь может получить список слотов.

### Фаза 6: Core-сервис: пользователь выбирает и играет в слот
**Цель:** Пользователь может подключиться к слоту и сделать спин.
**Фичи:** F-012, F-013
**Сущности:** —
**Результат:** Пользователь подключился к слоту и отправил запрос на спин.

### Фаза 7: Slot-сервис: обработка запросов на спины
**Цель:** Сервис получает запросы на спины, считает результат и отправляет изменения баланса.
**Фичи:** F-012, F-013
**Сущности:** VersionSpinStatistics
**Результат:** После того, как пользователь отправил запрос на спин, слот его обработал и отправил запрос на изменение баланса.

### Фаза 8: Core-сервис: отключить, включить слот
**Цель:** Администратор может отключить и включить слот.
**Фичи:** F-014, F-015
**Сущности:** —
**Результат:** Администратор отключил слот — к нему больше нельзя делать запросы на спины. Когда включил — стало можно.

## 6. Принципы и ограничения

### Архитектурные принципы
- Асинхронный код.
- Взаимодействие между сервисами должно быть очень быстрым: gRPC.
- Сервисы должны быть быстрыми: предпочтителен кэш для извлечения информации. В первую очередь кэшируются данные о слотах и активная версия слота.
- Кэш активной версии слота инвалидируется при её смене: изменение пишется в БД, затем в кэш.
- Сервисы должны быть быстрыми: оптимизированные алгоритмы.
- В отношении баланса — строгое Strong Consistency и высокая надёжность. Баланс изменяется после завершения спина (ставка + выигрыш = итоговое изменение баланса).

### Открытые вопросы
- Механизм консистентности кэша и БД при смене активной версии слота пока не определён.
- Механизм идемпотентности запроса на спин и синхронизация вычислений между Slot-сервисом и Core-сервисом пока не определён.
- Механизм предотвращения дублирующейся регистрации одного и того же клиента пока не определён.
- Механизм создания и управления учётной записью администратора пока не определён.

### Чего НЕ делаем (в рамках MVP)
- UI — система работает на уровне запросов.
- Мониторинг.
- Мультивалютность — баланс представлен просто числом без привязки к валюте.

### Технические предпочтения
- Python, FastAPI, gRPC.
